## Understanding the Gate-Level Netlist

Your gate-level netlist has some important characteristics:

### Dual-Rail Masking
The netlist was synthesized with boolean masking, which means:
- Every signal is split into two: signal_s0 and signal_s1
- The actual value is: `signal = signal_s0 XOR signal_s1`
- This provides protection against side-channel attacks

### Example Signal Mapping:
```
Standard Ibex:           Gate-Level Netlist:
--------------           -------------------
instr_req_o       →      instr_req_o_s0 + instr_req_o_s1
data_addr_o[31:0] →      data_addr_o_s0[31:0] + data_addr_o_s1[31:0]
```

### Random Bits Input:
The netlist has an additional input `randbits[15:0]` that provides randomization for the masking. This is generated by the GNG (Gaussian Noise Generator) module.

## Measuring Interrupt Latency

For Goal 2 (measuring interrupt latency with pre-synthesis RTL):

### Method 1: Use Traces
The tracer module logs all instruction execution. After simulation:
```bash
grep -E "irq|interrupt|mtvec" trace_*.log
```

### Method 2: Add Custom Monitor
Create a SystemVerilog monitor that tracks:
1. Interrupt assertion time (when irq_*_i goes high)
2. Handler entry time (when PC jumps to interrupt vector)
3. Calculate latency = entry_time - assertion_time

Example locations to probe:
```
irq_timer_i       - Interrupt input
dut.u_ibex_top.u_ibex_core.pc_id_i - Program counter
dut.u_ibex_top.u_ibex_core.csr_mtvec - Interrupt vector
```

### Method 3: Use Directed Interrupt Tests
The Ibex DV environment includes interrupt-specific tests:
```bash
./run_modified_rtl_sim.sh +UVM_TESTNAME=core_ibex_irq_test
```

## Troubleshooting

### Issue: Undefined modules
**Error:** `Unknown module 'XYZ'`
**Solution:** Add the missing module to `stdcell_models.v`

### Issue: X propagation
**Symptom:** Simulation shows X values
**Solution:** 
- Check reset timing
- Verify all inputs are driven
- Use `+delay_mode_zero` for gate-level

### Issue: Simulation hangs
**Solution:**
- Check clock generation
- Verify reset sequence
- Add timeout in testbench

### Issue: Assertion failures
**Symptom:** UVM assertion failures in gate-level but not RTL
**Solution:**
- Some assertions may need to be disabled for gate-level
- Check timing-related assertions
- Use `+DV_ASSERT_CTRL_DISABLE` to disable specific assertions

## Architecture Notes

### Testbench Hierarchy:
```
core_ibex_tb_top (testbench)
  ├── clk_rst_if (clock/reset interface)
  ├── irq_if (interrupt interface)
  ├── ibex_mem_intf (instruction memory)
  ├── ibex_mem_intf (data memory)
  ├── gng (random number generator)
  └── dut: ibex_top_tracing
      └── u_ibex_top
          └── u_ibex_core (← This is what the gate-level netlist implements)
```

For gate-level simulation, the wrapper makes `ibex_core_gatelevel_wrapper` look like a standard `ibex_core` to the rest of the hierarchy.
